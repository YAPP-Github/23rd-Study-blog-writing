## 개발자 글쓰기 내용 정리

### 프롤로그

- 정확성, 간결성, 가독성이 좋은 글을 쓰면 좋지만, 이 세가지 요소는 서로 대치되는 성질이 있다. 예를들면 간결성, 가독성이 올라가면, 정확성이 떨어질 수 있다. 그렇다면, 정확성, 간결성, 가독성을 두루 충족하는 글을 어떻게 작성할 수 있을까?
- 빅테크 기업에 입사하더라도 개발자 글쓰기를 배울 수 있는 환경은 많지 않은 것 같다. 수십 년 동안 다양한 개발 방법과 개발 도구가 생겨났지만 말이다.
  - 글쓰기도 과학이고 기술이다. 어떻게 개발자로서 좋은 글을 쓸 수 있을지 알아보자

### 1장, 개발자가 알아야할 글쓰기 기본

#### 문장과 단락의 구조화

문장을 쉽게 사용하기 위해서는 문장 구조로 핵심을 먼저 말한 뒤, 필요에 따라 부가설명을 한다. 컴퓨터는 개발자의 코드를 처음부터 끝까지 순서대로 순식간에 체계를 잡고 이해한다. 하지만, 사람은 계층이 없는 문서를 보면서 체계를 잡아내기 어렵다. 따라서, 계층을 표현하고, 단락을 구분해서 이해하기 쉽도록 문서를 작성하는게 중요하다.

- 서술식, 개조식, 도식
  - 서술식 : 
    - ~다. 로 끝나는 것. 줄거리가 있는 경우 사용 가능.
  - 개조식 : 
    - ~개선, ~완료, ~했음.
    - 줄거리가 없고, 중복되는 내용이 많을 경우 사용, 표를 통해 각 항목의 차이를 분명하게 표현 가능
    - 서술식에서 강조가 필요한 내용이 있다면 사용 가능

  - 도식 : 표, 도표로 보여주는 것

- 말머리 기호
  - 하위 요소로 갈 수록 기호가 더 작아지고, 들여쓰기하는 것이 좋다.
  - 문제를 서술하는 내용은 말머리 기호를 사용할 필요가 없다.
  - 순서를 표시하는 경우에는 말머리 기호를 번호로 붙인다.

- 단락을 구조화하는 방법
  - 개발자들은 단락을 맞추기 위해서 스페이스를 사용하는 경우가 많은데 이는 중간 들여쓰기가 안맞게 되는 주 원인이다.
    - 들여쓰기 시에는 스페이스 대신 탭을 활용하자. 탭을 활용하면 줄이 어긋나는 경우가 줄어든다.
  - 단락을 구분할때 글자 굵기, 모양, 줄 간 거리, 폰트 등을 함께 구분할 수 있다.

#### 쉽게 쓰는 띄어쓰기와 문장 부호

개발자가 어려워하는 것 중 하나, 띄어쓰기다. 어떤 상황에서 띄어쓰기를 할지, 개행을 할 지 고민될때가 많다. 언제 띄워쓰기와 개행을 하면 좋을지 알아보자!

- 조사는 모두 앞 낱말에 붙인다. (arg1, arg2, ...)
- 순서나 관계를 나타낼 때에는 뒤 낱말과 붙인다. (1단계, 2단계)
- 큰따옴표, 작음따옴표의 사용 예시
  - 큰따옴표 : c언어의 문자열, 책의 제목이나 신문 이름
  - 작은따옴표
    - 소제목이나 예술 작품의 제목, 상호, 법류, 규정 등을 나타낼때
    - 내용 강조 혹은 드러내고 싶을때

#### 영어 단어 선택과 외래어 표기법

개발을 하다보면 많은 영단어를 사용해야하는데, 이때 단어의 의미와 역할을 잘 이해하고 사용하는 것이 좋다.

- 멈추는 것과 관련된 영단어들
  - stop : 잠시 중단하는 것으로 언제든 다시 시작할 수 있을때
  - pause : 잠시 중단하는 것인데 금방이라도 다시 시작할 수 있는 상태
  - end : 끝나고, 재시작할 가능성이 전혀 없을때
  - finish : 끝장을 보았으므로, 재시작 고려할 필요조차 없는 상태
  - suspend : 다음 단계의 시작을 중단한 것
  - hold : 어떠한 의도로 인해 중단한 것

- 생성하는 것과 관련된 영단어들
  - create : 틀을 만들때 사용

- 수정하는 것과 관련된 영단어들
  - change : 내용을 단순히 바꿀때
  - modify : 잘못된 것을 바로잡을때
  - revise : 기존에 없던 새로운 정보나 아이디어를 붙여서 기존 내용과 달라짐을 명확히 하고 싶을때

- 반드시, 필수적인 의미의 영단어들
  - must : 필수 요구 사항
  - should : 권고 또는 권장 사항, 가능하면 지키거나 구현해야한다. (만약 구현이 어렵다면 다른 방법을 취할 수 있다.)
    - Q. -> 반드시 해야하는 경우에 사용되는 flag 값이 있다면 shouldPlay~ 보다는 mustPlay 등을 사용해야 할까?
